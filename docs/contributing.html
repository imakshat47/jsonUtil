ü§ù Contributing to JSON Utils

We welcome contributions! Whether you're fixing bugs, adding features, improving documentation, or sharing feedback, your help makes JSON Utils better for everyone.

üöÄ Getting Started

1. Fork & Clone

# Fork the repository on GitHub, then:
git clone https://github.com/YOUR_USERNAME/jsonutils.git
cd jsonutils
git remote add upstream https://github.com/originaluser/jsonutils.git

2. Set Up Development Environment

# Install dependencies
npm install

# Copy environment template
cp .env.example .env

# Edit .env with your development keys
# API_KEY=dev-api-key-123
# SYSTEM_KEY=dev-system-key-456

# Start the development server
npm run dev  # or npm start

3. Run Tests

# Run existing tests
npm test

# Run tests in watch mode during development
npm run test:watch

üîÑ Development Workflow

1. Create a Feature Branch

git checkout -b feature/your-feature-name
# or
git checkout -b fix/bug-description

2. Make Your Changes

  • Write clean, readable code
  • Follow existing code style and conventions
  • Add tests for new functionality
  • Update documentation when needed

3. Test Your Changes

# Run all tests
npm test

# Test specific functionality manually
curl -X POST http://localhost:4000/v1/forms \
  -H "Authorization: Bearer dev-api-key-123" \
  -H "X-Operation: FORM" \
  -H "X-Response-Type: html" \
  -H 'X-JSON-Payload: {"test":"value"}'

4. Commit & Push

git add .
git commit -m "feat: add new CSS framework support"
git push origin feature/your-feature-name

5. Open a Pull Request

Go to GitHub and create a pull request with:

  • Clear description of changes
  • Link to related issues
  • Screenshots for UI changes
  • Test results or examples

üìã Coding Standards

Code Style

  • Use ES modules (import/export)
  • Prefer const/let over var
  • Use descriptive variable names
  • Keep functions small and focused
  • Add JSDoc comments for complex functions

File Organization

  • Group related functionality
  • Use clear, consistent file naming
  • Keep configuration separate from logic
  • Add comments for complex sections

Testing

  • Write tests for new features
  • Test edge cases and error conditions
  • Use descriptive test names
  • Mock external dependencies

Security

  • Never commit secrets or keys
  • Validate all input data
  • Handle errors gracefully
  • Document security implications

üìù Commit Message Guidelines

Use conventional commit messages for better change tracking:

feat: add Tailwind CSS framework support
fix: resolve form generation for nested objects
docs: update API examples in README
test: add validation tests for edge cases
refactor: improve error handling in logger
chore: update dependencies to latest versions

Format:

type(scope): description

[optional body]

[optional footer]

Types:

  • feat: New feature
  • fix: Bug fix
  • docs: Documentation changes
  • style: Code style changes (formatting, etc.)
  • refactor: Code refactoring
  • test: Adding or updating tests
  • chore: Maintenance tasks
